# Zoom Out & Rethink

**当局部修补无法解决问题时，退后一步，重新审视问题本身。**

## 核心原则

反复修补却无效，往往不是执行力的问题，而是**问题定义错了**。你在水面下堵漏洞，而船底的钢板需要更换。

```
digraph stuck_detection {
    rankdir=TB;
    node [shape=box, style=rounded];
    
    same_issue [label="同一问题讨论 ≥3 轮？"];
    real_progress [label="每次修改后\n问题实质解决了吗？"];
    surface_fix [label="是否在治标不治本？\n（补丁套补丁）"];
    
    same_issue -> real_progress [label="是"];
    real_progress -> surface_fix [label="没有"];
    surface_fix -> "触发本技能" [label="是"];
}
```

## 诊断清单

遇到停滞时，依次回答以下问题：

### 第一步：我们解决的是真正的问题吗？

| 信号 | 含义 | 行动 |
|------|------|------|
| 修了 A，B 坏了；修了 B，A 又坏了 | 问题被错误地拆分，它们本质是同一个问题 | 合并看待，找到共同根因 |
| 每次修都引入新问题 | 当前架构无法优雅地容纳这个需求 | 退到架构层重新设计 |
| 讨论「怎么做」多于「做什么」 | 目标还没定义清楚就跳进了实现 | 回到需求定义 |
| 反复调整参数/阈值/边界条件 | 核心算法或数据模型选错了 | 考虑换方案而非调参 |

### 第二步：跳出到全局视角

**三个层次的思考：**

1. **当前层**：我们在讨论的具体问题 → "这个字段怎么映射？"
2. **架构层**：这个问题的上下文环境 → "这两个模块的边界划分对吗？"
3. **终态层**：理想状态下，这件事应该是什么样子 → "用户最终需要什么？从终态倒推，现在的设计合理吗？"

**实践方法：** 假设你是新加入团队的高级架构师，第一次看到这个问题。你会怎么设计？不受现有代码和之前讨论的约束。

### 第三步：评估问题本身

| 质问 | 目的 |
|------|------|
| **这个问题真的存在吗？** | 确认不是假设性问题或边缘 case |
| **这个问题值得解决吗？** | ROI 评估，可能直接接受现状更合理 |
| **能否简化需求来规避？** | 80% 的场景是否已经够用？ |
| **能否换个角度解决？** | 不解决 A，而是让 A 不再被需要 |
| **当前阻碍的根本原因是什么？** | 是技术债？架构限制？需求矛盾？ |

### 第四步：重新规划路径

如果确认问题需要解决，但当前路径走不通：

```
digraph rethink_paths {
    rankdir=LR;

    path1 [label="路径 1: 分步推进\n把问题拆成更小的步骤\n每步可独立验证"];
    path2 [label="路径 2: 架构重构\n先改骨架再填内容\n可能需要中间态"];

    stuck -> {path1 path2 path3 path4 path5};
}
```

## 常见陷阱

| 陷阱 | 表现 | 解法 |
|------|------|------|
| ** sunk cost ** | "已经改了这么多，不甘心推倒重来" | 沉没成本不是成本。问：如果从零开始，你会选当前方案吗？ |
| **细节沉迷** | 越来越深地讨论某个实现细节 | 强制回退到第三步：这个问题在终态视角下重要吗？ |
| **虚假进度** | 每次都有 "进展" 但问题仍在 | 区分"做了事"和"解决了问题"，进展 ≠ 前进 |
| **问题膨胀** | 试图一次性解决所有相关问题 | 缩小范围，先解决核心矛盾 |
| **方向性死磕** | 坚信当前方向一定能走通 | 设定止损点：如果再试 N 次还是不行，就必须换方向 |

## 自检红牌

以下信号出现时，**立即停止当前方向的讨论**，执行本技能的完整分析流程：

- 🚫 连续 3 次修改后问题仍然存在
- 🚫 开始讨论 "特殊情况" 和 "边界条件" 越来越多
- 🚫 某个方案需要大量例外处理和特殊逻辑
- 🚫 "如果...就..." 句式出现超过 5 次
- 🚫 自己都觉得方案 "丑" 或 "不够优雅"
- 🚫 为了解决一个问题引入了两个新问题
- 🚫 用户表达了 "怎么又回来了" 或 "上次不是改过了吗" 的意思

## 适用场景与不适用场景

**适用：**
- 架构设计反复调整无法定稿
- Bug 修了又出现，或在别的地方冒出来
- 性能优化陷入瓶颈
- API/接口设计反复争论
- 重构过程中发现 "越改越乱"
- 需求讨论陷入 "既要又要还要"

**不适用：**
- 明确的 bug，根因已定位，只是在调试
- 简单的配置或环境问题
- 纯执行类任务（写 CRUD、改样式等）
- 用户明确要求按当前方案继续
