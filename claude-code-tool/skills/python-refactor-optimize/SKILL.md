# Python Refactor Optimize Skill

用于在执行完 python-refactor 重构后，将本次执行过程中踩的坑记录到 negative-case 用例库。

## 何时触发

以下情况应该触发此 skill：

1. **重构脚本执行报错** - python-refactor 脚本运行失败
2. **重构结果不符合预期** - 移动/重命名后导入未更新、文件位置错误等
3. **发现 rope 限制** - 某些操作 rope 无法完成或效果不佳
4. **用户明确要求** - 用户说"记录这个坑"、"添加到负向用例库"
5. **重构完成后的复盘** - 用户说"总结这次重构的问题"

## 核心工作流程

### Step 1: 收集问题信息

向用户询问以下信息（如果用户未主动提供）：

1. **问题摘要**：用简短的中英文描述问题核心
   - 示例：`MoveModule不更新不匹配的导入路径`、`get_source_folders性能问题`

2. **错误现象**：
   - 完整的错误信息或错误现象描述

3. **错误脚本**：
   - 导致错误的脚本内容（如果有）

4. **相关代码**：
   - 涉及的文件结构和关键代码片段

5. **根本原因**（如果已知）：
   - 为什么会出现这个问题

6. **解决方案**（如果已找到）：
   - 如何解决这个问题

### Step 2: 创建用例目录

根据模板创建用例目录结构：

```
negative-case/raw-case/{问题摘要}/
├── analysis.md          # 问题分析和解决方案（必需）
├── case_script.py      # 出错的脚本（如果有）
├── case_output.txt     # 完整的错误输出（如果有）
└── case_src/          # 复现问题的最小源代码（可选）
```

**注意**：
- 不需要检查是否已存在类似问题，直接创建新目录
- 目录名使用简短的中英文描述问题核心
- 即使问题尚未解决，也可以先记录错误现象，之后补充解决方案

### Step 3: 写入 analysis.md

按照以下模板填写：

```markdown
# {问题摘要}

## 问题概述

简要描述遇到的问题（一句话）。

## 错误现象

```
粘贴错误信息或描述错误现象
```

## 根本原因

分析错误的根本原因。

## 解决方案

描述如何解决这个问题（如果已知）。

## 经验总结

- 关键词1：总结1
- 关键词2：总结2

## 相关文件

- `case_script.py` - 出错的脚本
- `case_output.txt` - 完整的错误输出
- `case_src/` - 复现问题的最小源代码
```

> 2. **错误信息**：完整的错误输出
> 3. **相关脚本**：导致错误的脚本（可以在消息中贴给我）

### 用户：我重构完了，有些问题想记录

**你**：
> 好的，让我们总结一下这次重构踩的坑。请告诉我：
>
> 1. 遇到了哪些问题？
> 2. 每个问题的错误现象是什么？
> 3. 是否已找到解决方案？
>
> 我会帮你逐一记录到负向用例库。

## 常见问题类型参考

根据已有的用例库，常见问题包括：

| 类型 | 示例 |
|------|------|
| 版本控制 | "not under version control" - 重命名模块需要 VCS |
| 性能问题 | get_changes 很慢 - 需要使用 resources 参数 |
| 路径问题 | MoveModule 不能同时改名 - 需要先 Rename 再 Move |
| 导入问题 | 不匹配的导入路径不会被更新 - 需要先修复导入 |
| AST 问题 | 类型注解导致 AST 位置不同步 |

## 注意事项

1. **不需要检查重复**：直接创建新目录，覆盖或新建都可以
2. **即使未解决也能记录**：可以先记录错误现象，后续再补充解决方案
3. **保留关键信息**：错误脚本、错误输出、相关代码都要保存
4. **提供解决方案**：如果问题已解决，务必记录解决方案供后人参考
